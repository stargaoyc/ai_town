二次元AI小镇陪伴智能体 — 去 MongoDB 设计文档
> 本文档基于 `1.md` 总体方案，移除 MongoDB，将全部持久化数据统一收敛到 PostgreSQL 17 + pgvector，并给出迁移映射、表结构、数据访问层、索引策略与权衡分析。

---

## 一、方案分析

### 1.1 为什么去除 MongoDB

| 维度 | 原 MongoDB 方案的问题 | 去 MongoDB 后的改善 |
|------|----------------------|---------------------|
| 一致性 | 角色定义(PG)与行为记录(Mongo)跨库，无跨库事务，状态/行为/记忆易出现"半写" | 全部落 PG，可在一个事务内完成 State→Action→Memory 闭环 |
| 运维 | 需独立部署、备份、监控 Mongo 与 PG 两套数据库 | 单一数据库栈，备份/恢复/升级路径唯一 |
| Schema 演进 | Mongo 无强约束，字段漂移靠应用层防御 | PG 提供 DDL 约束 + JSONB 兼顾灵活字段，迁移可被版本化(`alembic`)管理 |
| 向量与结构分离 | 行为记录在 Mongo、向量在 Chroma/Milvus，查询需两阶段 join | pgvector 直接在 MemoryEpisode 表存向量，单 SQL 完成"过滤+检索+取详情" |
| 连接池 | motor 与 asyncpg 两套连接池，资源与超时分别治理 | 单一 asyncpg/SQLAlchemy 池，配置收敛 |
| 测试 | 需起 Mongo + PG + Redis + 向量库四个依赖 | 需 PG + Redis 两个依赖，CI 更轻 |

### 1.2 可行性结论

- 原方案中 MongoDB 的 6 个集合**全部为结构化或半结构化数据**，无深嵌套文档、无 MapReduce 聚合管道需求，可无损映射到 PG 表 + JSONB。
- `MemoryEpisode` 原本依赖独立向量库，迁移到 `pgvector` 后，HNSW 索引在 10 万级记忆量下 p95 检索 < 30ms，满足角色 Tick 实时性。
- 消息历史 `Message` 在 PG 中分区表 + BRIN 索引可承载千万级消息。

### 1.3 迁移映射总表

| 原 MongoDB 集合 | 新 PG 表 | 主键 | 关键字段处理 | 备注 |
|-----------------|----------|------|--------------|------|
| `Character` | `characters` | `id UUID` | `traits JSONB`、`personality TEXT[]` | 替代 Mongo 文档 |
| `ActionRecord` | `action_records` | `id UUID` | `params JSONB`、`related_characters UUID[]` | 按 `created_at` 月分区可选 |
| `Reflection` | `reflections` | `id UUID` | `summary TEXT`、`source_memory_ids UUID[]` | — |
| `Plan` | `plans` | `id UUID` | `steps JSONB`、状态枚举 | — |
| `Message` | `messages` | `id UUID` | `metadata JSONB`，按 `created_at` 分区 | 千万级友好 |
| `Relation` | `relations` | `(from_id, to_id)` 复合主键 | `strength INT`、`tags TEXT[]` | 唯一约束防重 |
| `MemoryEpisode`(向量库) | `memory_episodes` | `id UUID` | `embedding vector(1536)` + HNSW 索引 | 合并 Chroma/Milvus |
| `ModuleConfig`(原 PG) | `module_configs` | `id UUID` | 不变 | 已是 PG |

```text
           │                                            │
           │            ┌───────────────┐              │
           └────────────│   Update     │◀──────────────┘
                        │  单一 PG 事务 │
```

**核心改进**：Action 执行后的"写行为记录 + 写记忆向量 + 更新反思状态"在**同一个 PG 事务**中完成，杜绝原方案跨库半写问题。

### 3.2 表结构 DDL

#### 3.2.1 扩展与基础设置

```sql
-- 启用扩展
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS "vector";       -- pgvector
CREATE EXTENSION IF NOT EXISTS "pg_trgm";      -- 文本模糊检索

-- 命名约定:
--   主键: id UUID DEFAULT uuid_generate_v4()
--   时间: created_at / updated_at TIMESTAMPTZ DEFAULT now()
--   软删除: deleted_at TIMESTAMPTZ (按需)
--   灵活字段: 用 JSONB, 不用 JSON
```

#### 3.2.2 characters（替代 Mongo: Character）

```sql
CREATE TABLE characters (
    id           UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name         TEXT NOT NULL,
    age          INT  NOT NULL CHECK (age >= 0 AND age <= 200),
    occupation   TEXT NOT NULL,
    personality  TEXT[] NOT NULL DEFAULT '{}',           -- ["开朗","细心"]
    traits       JSONB NOT NULL DEFAULT '{}'::jsonb,      -- 自定义属性
    backstory    TEXT NOT NULL DEFAULT '',
    avatar_url   TEXT,
    status       TEXT NOT NULL DEFAULT 'active'
                 CHECK (status IN ('active','archived','deleted')),
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_characters_name_trgm ON characters USING gin (name gin_trgm_ops);
CREATE INDEX idx_characters_traits    ON characters USING gin (traits jsonb_path_ops);
CREATE INDEX idx_characters_status    ON characters (status);
```

#### 3.2.4 action_records（替代 Mongo: ActionRecord，按月分区）

```sql
CREATE TABLE action_records (
    id                 UUID NOT NULL DEFAULT uuid_generate_v4(),
    character_id       UUID NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
```

#### 3.2.5 memory_episodes（替代独立向量库：MemoryEpisode）

```sql
CREATE TABLE memory_episodes (
    id                 UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    character_id       UUID NOT NULL REFERENCES characters(id) ON DELETE CASCADE,
    content            TEXT NOT NULL,
    embedding          vector(1536) NOT NULL,            -- OpenAI text-embedding-3-small
    importance         INT  NOT NULL DEFAULT 5 CHECK (importance BETWEEN 1 AND 10),
    timestamp          TIMESTAMPTZ NOT NULL DEFAULT now(),
    action_id          TEXT,
    location           TEXT,
    related_characters UUID[] NOT NULL DEFAULT '{}',
    -- 反思衍生标记
    is_reflected       BOOLEAN NOT NULL DEFAULT FALSE,
    source_type        TEXT NOT NULL DEFAULT 'action'
                       CHECK (source_type IN ('action','conversation','reflection','event'))
);

-- HNSW 向量索引 (检索性能优于 IVFFlat, 适合生产)
CREATE INDEX idx_mem_embedding_hnsw
    ON memory_episodes
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- 过滤+检索联合索引 (角色内检索最常见)
CREATE INDEX idx_mem_char_time ON memory_episodes (character_id, timestamp DESC);
CREATE INDEX idx_mem_char_imp   ON memory_episodes (character_id, importance DESC);
CREATE INDEX idx_mem_related    ON memory_episodes USING gin (related_characters);
```

### 3.3 索引策略汇总

| 检索场景 | 索引 | 说明 |
|----------|------|------|
| 角色名模糊搜索 | `gin (name gin_trgm_ops)` | pg_trgm 支持相似度 |
| 角色按 traits 筛选 | `gin (traits jsonb_path_ops)` | JSONB 路径查询 |
| 角色记忆向量召回 | `hnsw (embedding vector_cosine_ops)` | HNSW，生产首选 |
| 角色内记忆按时间 | `(character_id, timestamp DESC)` | 范围扫描 |
| 角色行为历史 | `(character_id, created_at DESC)` | 分区裁剪 + 索引 |
| 关联角色反查 | `gin (related_characters)` | 数组成员查询 |
| 关系图遍历 | `(from_id)` PK + `(to_id)` 索引 | 双向查询 |
| 消息会话拉取 | `(conversation_id, created_at)` | 多轮对话上下文 |
| 消息时间扫描 | `brin (created_at)` | 分区表海量时间数据 |

### 3.4 向量检索 SQL 范式

```sql
-- 角色记忆 Top-K 检索 + 时间衰减重排序
SELECT id, content, importance, timestamp,
       1 - (embedding <=> :q_vec) AS sim_score
FROM memory_episodes
WHERE character_id = :cid
ORDER BY embedding <=> :q_vec
LIMIT :top_k;

-- 反思层语义检索 (高层认知)
SELECT id, summary
FROM reflections
WHERE character_id = :cid
ORDER BY embedding <=> :q_vec
LIMIT 5;

-- 联合检索 (记忆 + 反思) 在应用层合并, 或用 UNION ALL
```

```python
# db/session.py
from sqlalchemy.ext.asyncio import (
    AsyncSession, async_sessionmaker, create_async_engine
)
from contextlib import asynccontextmanager

class DB:
    def __init__(self, url: str, pool_size: int = 20, max_overflow: int = 10):
        # url 形如: postgresql+asyncpg://user:pass@host:5432/ai_town
        self.engine = create_async_engine(
            url,
            pool_size=pool_size,
            max_overflow=max_overflow,
            pool_pre_ping=True,
            echo=False,
        )
        self.session_factory = async_sessionmaker(
            self.engine, class_=AsyncSession, expire_on_commit=False
        )

    @asynccontextmanager
    async def session(self) -> AsyncSession:
        async with self.session_factory() as s:
            try:
                yield s
                await s.commit()
            except Exception:
                await s.rollback()
                raise
```

### 4.4 Repository 模式示例

```python
# db/repositories/memory_repo.py
from sqlalchemy import select, desc
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy.orm import selectinload
from db.models.memory_episode import MemoryEpisode
import uuid

class MemoryRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def add(self, ep: MemoryEpisode) -> MemoryEpisode:
        self.session.add(ep)
        await self.session.flush()
        return ep

    async def search_similar(
        self,
        character_id: uuid.UUID,
        query_vec: list[float],
        top_k: int = 10,
    ) -> list[MemoryEpisode]:
        stmt = (
            select(MemoryEpisode)
            .where(MemoryEpisode.character_id == character_id)
            .order_by(MemoryEpisode.embedding.cosine_distance(query_vec))
            .limit(top_k)
        )
        result = await self.session.execute(stmt)
        return list(result.scalars())

    async def recent(self, character_id: uuid.UUID, limit: int = 50):
        stmt = (
            select(MemoryEpisode)
            .where(MemoryEpisode.character_id == character_id)
            .order_by(desc(MemoryEpisode.timestamp))
            .limit(limit)
        )
        return list((await self.session.execute(stmt)).scalars())
```

### 4.6 alembic 迁移要点

- HNSW 索引不能通过 ORM `Index` 自动生成，需在 alembic 升级脚本中用 `op.execute()` 写原生 SQL：

```python
# migrations/versions/xxxx_init.py
def upgrade():
    op.execute("CREATE EXTENSION IF NOT EXISTS vector;")
    # ... 建表 ...
    op.execute(
        "CREATE INDEX idx_mem_embedding_hnsw ON memory_episodes "
        "USING hnsw (embedding vector_cosine_ops) "
        "WITH (m = 16, ef_construction = 64);"
    )
```

- 分区表按月预创建未来 12 个月分区，配合定时任务(`pg_cron` 或应用层)滚动新建。

## 五、Redis 角色调整

去 MongoDB 后，Redis 职责更聚焦：

| 用途 | 数据结构 | TTL/策略 |
|------|----------|----------|
| 角色实时状态 | Hash `char:{id}:state` | 永久 + 30s 写回 PG |
| 世界实时状态 | Hash `world:state` | 永久 + 60s 写回 PG |
| 角色锁 (Tick 互斥) | String `char:{id}:lock` | 30s 自动过期 |
| 限流 | 滑窗/令牌桶 | 按需 |
| 消息队列 | Redis Streams `mq:actions`、`mq:push` | 消费组 ACK |
| 热点记忆缓存 | String(JSON) `mem:{id}:recent` | 5min |
| 会话上下文 | List/Stream `conv:{id}` | 1h 空闲过期 |

**冷启动**：服务起来后从 PG `character_states` / `world_snapshots` 一次性灌入 Redis。

## 七、迁移方案（如已有 Mongo 数据）

### 7.1 一次性迁移脚本

```python
# scripts/migrate_mongo_to_pg.py
# 顺序: characters → relations → action_records → memory_episodes
#       → reflections → plans → messages → module_configs
```

要点：
1. **依赖顺序**：先无外键的 `characters`，再带外键的其余表。
2. **批量写入**：每批 1000 行，`COPY` 优于 `INSERT`。
3. **向量回填**：`MemoryEpisode` 迁移时若 Mongo 中无原始向量，需按 `content` 调 embedding 接口重算。
4. **校验**：逐集合对比行数、关键 ID 抽样比对内容。
5. **回滚**：迁移期间双写（旧 Mongo 仍写），出错可切回。

### 7.2 数据量估算

| 表 | 单角色/月增量 | 50 角色年增量 | 说明 |
|----|---------------|---------------|------|
| action_records | ~3 万 | ~1800 万 | 按 30s/Tick, 每角色每 Tick 1 条 |
| memory_episodes | ~3 万 | ~1800 万 | 与 action_records 1:1 |
| messages | 视用户量 | — | 分区表承载 |
| reflections | ~50 | ~3 万 | 反思稀疏 |

PG 在 HNSW + 分区表下，单表千万级行检索 p95 < 50ms（HNSW 检索 + 过滤）。

## 八、优势与权衡

### 8.1 优势

1. **事务一致性**：状态/行为/记忆同一事务，闭环不再"半写"。
2. **栈简化**：从 4 套基础设施降到 3 套，部署/备份/监控成本下降。
3. **Schema 治理**：alembic 版本化迁移，约束防字段漂移。
4. **检索表达力**：单 SQL 完成"过滤 + 向量召回 + JSONB 字段提取 + 关联 join"。
5. **可观测**：PG 自带 `pg_stat_statements`、慢查询日志，问题定位比 Mongo 直观。
6. **测试/CI**：依赖减少，启动更快。

### 8.2 权衡与对策

| 权衡 | 风险 | 对策 |
|------|------|------|
| pgvector 检索性能在亿级向量下不如专用向量库 | 量级爆炸后退化 | 当前 10M 级 HNSW 足够；预留接口可平滑切到 Milvus |
| JSONB 写入比 Mongo 文档略慢 | 高频写场景 | `action_records` 走分区 + 批量写；热数据先 Redis |
| 单点 PG 故障影响面更大 | 数据库宕机 | 流复制 + Patroni 高可用 + 异地备份 |
| Mongo 聚合管道能力缺失 | 复杂分析 | 用 SQL 窗口函数 + 物化视图替代 |
| 分区表需滚动维护 | 运维负担 | `pg_cron` 自动预创建未来分区 |

### 8.3 何时该重新引入向量库

满足任一条件时，建议把 `memory_episodes` 迁回独立向量库（如 Milvus），PG 仅存元数据：
- 单角色记忆数 > 500 万，或总记忆数 > 1 亿；
- HNSW 索引构建内存占用超过 PG `shared_buffers` 50%；
- 检索 p95 > 200ms 且调参无效。

接口层 `MemoryRepository` 已抽象，切换成本仅限实现类。
